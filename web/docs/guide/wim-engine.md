# 镜像引擎（libwim / wimgapi）

LetRecovery 释放和捕获 WIM / ESD 镜像时，可以在两套引擎之间选择。一般用户保持默认即可，
本页解释它们的区别，方便有需要时切换。

## 两套引擎

| 引擎 | 来源 | 特点 |
| --- | --- | --- |
| **libwim** | 内置的 `libwim-15.dll` | **默认**。跨环境一致，桌面与 WinPE 行为相同，不依赖系统组件。 |
| **wimgapi** | 系统自带的 `wimgapi.dll` | Windows 原生镜像 API。可选，部分场景下与系统行为更贴合。 |

## 在哪里切换

在**关于**页选择镜像引擎。该选择是**进程级全局**的：

- 正常系统端从 `config.json` 读取并设置；
- 走[经 PE 安装/备份](/guide/system-install#安装方式)时，所选引擎会**随重启传递给 PE 端**——
  也就是说"切到 wimgapi 后，PE 端也用 wimgapi"，前后一致。

## 自动回退，永不卡死

选择 wimgapi 后并不会"孤注一掷"：

- 若 wimgapi **加载/初始化失败**，构造时就**自动回退 libwim**；
- 若 wimgapi 在**释放或捕获过程中失败**，也会回退到 libwim 重试（捕获失败回退前，会先清理
  wimgapi 留下的半成品文件，避免被误当成增量追加）。

libwim 始终会被初始化，作为主引擎或回退引擎，因此功能**始终可用**。

::: tip 该用哪个？
拿不准就用**默认的 libwim**——它跨环境一致、最稳。只有在你明确遇到某个镜像用 libwim 处理
异常、想试试系统原生 API 时，才需要切到 wimgapi。
:::

::: details 一个只读例外
探测"镜像里是否内置应答文件"这类**只读元数据**操作始终走 libwim（仅读、不挂载），与
释放/捕获的引擎选择无关。
:::
